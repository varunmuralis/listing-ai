# ListingAI — Engineering Handoff

Read this first if you're picking up the build. Pairs with `CURRENT_STATE.md`
(what exists) and `IMPLEMENTATION.md` (the original plan).

## How to run

```bash
npm install
npm run dev            # http://localhost:3000  (no credentials needed)
npx vitest run         # unit + integration tests
npm run build          # production build (typecheck + lint)
npx next typegen       # regenerate PageProps route types if TS complains
```

No `.env` is required locally — the app falls back to in-memory persistence, dev
cookie auth, a grounded mock AI, and intentional "unavailable" states for the map.

## Architecture decisions (and why)

1. **Next.js 16 specifics.** `params`/`searchParams`/`cookies()`/`headers()` are
   async. Middleware is renamed to **`proxy.ts`**. `PageProps<'/route'>` is a
   global helper generated by `next typegen`. Server Actions for mutations, Route
   Handlers for polling/streaming.
2. **Swappable persistence + auth.** This sandbox has no Docker, so a local
   Supabase stack can't run. Everything talks to repository/auth *interfaces*.
   `getRepositories()` returns the Supabase adapter when `NEXT_PUBLIC_SUPABASE_URL`
   + anon + service-role keys are present, otherwise the in-memory adapter. This
   is a persistence-adapter swap, **not** fabricated data — listing data still
   flows through the real provider → processing → repository pipeline. When you
   get a Supabase project, set the env vars and run the migrations; no app code
   changes needed.
3. **Mock only at the external boundary.** `ListingDataProvider` is the single
   entry point for listing data; **no Zillow scraping**. The mock returns a
   deterministic fixture (`src/lib/providers/listing/fixtures/`). Swap in a
   licensed MLS adapter behind the same interface.
4. **Real processing, persisted progress.** `processing-service.ts` runs actual
   steps through the provider/classifier/storage boundaries and writes progress
   to the `processing_jobs` row after each step. The UI polls that row via
   `useJobStatus` (encapsulated so it can become Supabase Realtime later). Steps
   are paced with short awaits to model latency — progress is derived from real
   state, not a fake timer. Idempotency: images dedupe by `(project_id, source_url)`,
   property is upserted, and a per-process guard prevents double-runs.
5. **Grounded AI.** The assistant only sees context assembled from the authorized
   project (`assistant-context.ts`) and is instructed to distinguish verified
   facts / estimates / recommendations / unavailable info, and never to invent
   school ratings, taxes, room dimensions, crime, or walk scores. The no-key mock
   provider enforces the same discipline deterministically.
6. **Authorization everywhere.** Every project read/mutation goes through
   `authorizeProject()`; owner ids come from the session, never the client. RLS
   is the second layer in production.

## Database schema (Supabase — `supabase/migrations/0001_init.sql`)

Enums: `project_status`, `job_type`, `job_status`, `processing_step`,
`room_type`, `message_role`.

| Table | Purpose | Key columns |
|-------|---------|-------------|
| `profiles` | 1:1 with `auth.users` | id, full_name, avatar_url |
| `projects` | a listing workspace | owner_id, source_url, title, status, progress, active_job_step, error_message |
| `properties` | 1:1 with project | address_line_1, city, region, postal_code, lat/long, price, bedrooms, bathrooms, square_feet, lot_size, year_built, hoa_monthly, annual_property_tax, description, amenities[] |
| `property_images` | imported photos | source_url, storage_path, sort_order, width, height, room_type, room_confidence, metadata; **unique (project_id, source_url)** for idempotency |
| `processing_jobs` | ingestion jobs | type, status, progress, current_step, attempts, payload, result, error_message, started_at, completed_at |
| `conversations` | 1 per (project, owner) | project_id, owner_id |
| `messages` | chat history | conversation_id, role, content, citations |

RLS: users reach only rows under projects they own (`owns_project()` helper).
Storage bucket `property-images` (0002) is public-read, owner-write.
`updated_at` maintained by triggers; a trigger auto-creates a profile on signup.

## Environment variables (`.env.example`)

All optional. Presence toggles the corresponding real provider.

| Var | Effect when set |
|-----|-----------------|
| `NEXT_PUBLIC_SUPABASE_URL` / `NEXT_PUBLIC_SUPABASE_ANON_KEY` / `SUPABASE_SERVICE_ROLE_KEY` | Use Supabase auth + Postgres + Storage instead of in-memory |
| `OPENAI_API_KEY` (+ optional `OPENAI_BASE_URL`, `OPENAI_MODEL`) | Use OpenAI streaming instead of the grounded mock |
| `NEXT_PUBLIC_MAPBOX_TOKEN` | Enable the Mapbox map (else "missing token" state) |
| `LISTING_PROVIDER` (`mock`\|`manual`) | Ingestion provider |
| `ROOM_CLASSIFIER` (`heuristic`\|`ai`) | Classifier |
| `APP_SESSION_SECRET` | Signs the dev-auth cookie |

## Known bugs / risks

- **Dev persistence is in-memory**: data resets on server restart and is
  single-process only. Fine for the dev slice; use Supabase for anything real.
- **Background job durability**: `ensureProcessing` fires-and-forgets within the
  request process. Works in `next dev`; in serverless it could be killed after
  the response. Production needs a real queue/worker (the `queue` boundary is
  the place to add it).
- **Supabase adapter is unexercised locally** — it typechecks but hasn't run
  against a live DB. Validate against a real project before trusting it.
- **Fixture images** come from `picsum.photos` (allowlisted in `next.config.ts`).
  They require network at render time; the map/AI do not.
- The **3D viewer** may be present from the background subagent but is not wired
  into a route yet — typecheck it before importing.
